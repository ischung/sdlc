# 이슈 분할 전략 비교 — 수직 슬라이스 vs 아키텍처 계층별

두 분할 전략은 "무엇을 기준으로 이슈를 쪼개는가"가 본질적으로 다르며, 프로젝트 성격에 따라 장단점이 뚜렷하게 갈린다.

본격적인 비교 전에, 수직 슬라이스 전략에서 전제로 삼는 **Walking Skeleton**과 **MVP** 개념을 먼저 정리한다.

---

## Walking Skeleton이란?

> **한 줄 정의**: 시스템을 구성하는 **모든 주요 조각을 실제로 한 번씩 관통시킨**, 가장 얇고 최소한의 End-to-End 동작.

이름 그대로 "걸어다니는 뼈대"다. 살(기능)은 하나도 붙지 않았지만, 골격이 전부 연결되어 있어 **실제로 걸을 수는 있는** 상태를 말한다.

### 비유로 이해하기

**✈️ 비행기 제작 비유**
- ❌ Walking Skeleton이 아님: 날개만 완벽하게 만들고, 동체는 아직, 엔진도 아직 → "아무것도 안 난다"
- ✅ Walking Skeleton: 아주 작은 종이비행기라도 **직접 던져서 날아간다**. 엔진·날개·꼬리가 모두 최소 형태로 존재하고 한 번은 비행한다.

**🏠 집 짓기 비유**
- ❌ Walking Skeleton이 아님: 거실만 완성, 부엌·침실·화장실은 아직 → "아직 살 수 없다"
- ✅ Walking Skeleton: 전기·수도·문·창문·지붕이 모두 연결된 **아주 작은 원룸**. 살림살이는 없지만, 일단 하룻밤 잘 수는 있다.

### 소프트웨어에서의 Walking Skeleton 예시

**🍕 피자 주문 앱**
- Frontend: "주문하기" 버튼 하나만 있는 빈 페이지
- Backend: 버튼 누르면 `POST /orders`로 하드코딩된 "페퍼로니 1판" 주문이 DB에 저장됨
- DB: `orders` 테이블에 레코드 1개 생김
- CI/CD: main 브랜치 푸시 → 프로덕션 URL에 자동 배포됨
- **완료 기준**: 누구든 실제 URL에 접속해서 버튼을 누르면 DB에 주문이 쌓이고, 다음 접속 시 그 내역이 보인다.

메뉴 선택도, 로그인도, 결제도, 배송도, 주문 취소도 **전혀 없다**. 하지만 "브라우저 클릭 → API → DB → 다시 화면"이라는 관통 경로는 **실제로 작동**한다.

**📝 블로그 서비스**
- Frontend: 제목/본문 입력 폼 하나 + 글 목록 페이지
- Backend: `POST /posts`, `GET /posts` 두 엔드포인트
- DB: `posts` 테이블
- **완료 기준**: 프로덕션 URL에서 아무나 글을 쓰고 목록에서 볼 수 있다.

로그인, 에디터, 이미지 업로드, 댓글, 검색, 카테고리 **전부 없다**.

### Walking Skeleton의 목적

- **통합 리스크를 가장 먼저 발견한다.** (CORS, 환경 변수, 빌드 파이프라인, 인증 토큰 등은 실제로 붙여봐야 드러나는 문제다.)
- **이후 모든 기능은 "이미 돌아가는 시스템 위에 얹기"가 된다.** → 첫 배포의 심리적 장벽이 사라진다.
- **팀/이해관계자에게 "일단 뭔가 돌아간다"는 것을 조기에 증명한다.**

## MVP란?

> **한 줄 정의**: **실제 사용자가 의미 있게 사용할 수 있는** 최소한의 기능 집합을 갖춘 제품. "Minimum Viable Product".

Walking Skeleton은 **기술적으로** 최소지만 **사용자에게는 가치가 없다**. MVP는 반대로, **사용자에게 진짜 가치를 주는** 가장 작은 제품이다.

### 비유로 이해하기 — 🛹 고전 이동수단 비유 (Henrik Kniberg)

> 사용자의 니즈가 "A 지점에서 B 지점으로 이동"인 경우, 자동차를 만드는 방법은 두 가지다.

**❌ MVP가 아닌 방식 (레이어별 점진적 조립)**

```
1주차: 바퀴만 → 2주차: 바퀴+차축 → 3주차: +차대 → 4주차: +차체 → 5주차: 완성
```

→ 마지막 주까지 사용자는 **아무 것도 타지 못한다.** 피드백을 받을 길이 없다.

**✅ MVP 방식 (각 단계마다 사용 가능)**

```
1주차: 스케이트보드 → 2주차: 킥보드 → 3주차: 자전거 → 4주차: 오토바이 → 5주차: 자동차
```

→ **1주차부터** 사용자는 실제로 이동할 수 있다. 스케이트보드가 불편하다는 피드백을 들으면 방향을 조정할 수 있다. 사용자는 매 단계마다 "더 나아진" 제품을 경험한다.

### 소프트웨어에서의 MVP 예시

**🎵 Spotify 초기 MVP (실제 사례)**
- 검색 기능 **있음**
- 재생 기능 **있음** (일부 장르, 수십만 곡)
- 플레이리스트 만들기 **있음**
- 추천 알고리즘 **없음**
- 모바일 앱 **없음** (데스크톱만)
- 소셜 기능 **없음**
- 팟캐스트 **없음**

→ 사용자가 "음악을 찾아서 들을 수 있다"는 핵심 가치는 전달된다. 나머지는 이후에 붙인다.

**🏠 Airbnb 초기 MVP (실제 사례)**
- 집주인이 자기 집 사진과 가격을 올릴 수 있음
- 여행자가 그 목록을 보고 연락할 수 있음
- 결제는 **이메일과 수기**로 처리
- 별점, 리뷰, 자동 결제, 지도 뷰 **전부 없음**

→ "남의 집에 묵는다"는 핵심 가치는 동작한다. 결제 자동화는 수요가 확인된 뒤에 붙였다.

### MVP의 목적

- **사용자 가치 가설을 최소 비용으로 검증한다.** ("사람들이 정말 이걸 원할까?"를 물어본다.)
- **조기 피드백 루프를 만든다.** 잘못된 방향을 2주 만에 발견하고 수정할 수 있다.
- **Sunk cost를 줄인다.** 풀 기능을 만든 뒤 "안 쓴다"는 걸 알면 너무 늦다.

## Walking Skeleton vs MVP — 같은 듯 다른 개념

| 구분 | Walking Skeleton | MVP |
|------|-----------------|-----|
| **관심사** | 기술 아키텍처의 E2E 연결 | 사용자 가치 검증 |
| **완료 기준** | 프로덕션 URL이 존재하고 뭔가 응답함 | 사용자가 실제로 유용하게 사용함 |
| **사용자에게 의미** | 거의 없음 ("그래서 뭘 할 수 있는데?") | 핵심 가치 하나를 완결되게 제공 |
| **순서** | MVP 이전 단계 (Phase 0) | Walking Skeleton 위에 기능을 얹어 도달 (Phase 1~2) |
| **비유** | 비행하는 종이비행기, 하룻밤 잘 수 있는 빈 원룸 | 출퇴근 가능한 스케이트보드 |

> **핵심**: Walking Skeleton은 **MVP로 가기 위한 출발점**이다.
> 먼저 뼈대를 한 번 관통시키고(Phase 0), 그 뼈대에 수직 슬라이스로 유저 스토리를 하나씩 붙여가며(Phase 1~2) MVP에 도달한다.

## 두 분할 전략의 차이 — 한눈에 보기

| 관점 | 수직 슬라이스 (Vertical Slice) | 아키텍처 계층별 (Horizontal / Layered) |
|------|-------------------------------|----------------------------------------|
| **분할 기준** | 유저 스토리 / 기능 단위 | 기술 레이어(DB, Backend, Frontend, Test 등) |
| **한 이슈의 관통 범위** | UI + API + DB + 테스트까지 전부 | 단일 레이어 한정 |
| **첫 배포까지 걸리는 시간** | 매우 짧음 (Walking Skeleton이 1번 이슈) | 모든 레이어 이슈가 끝나야 배포 가능 |
| **사용자 가치 전달 시점** | 각 이슈 완료 즉시 | 전체 이슈 완료 후 |
| **리스크 발견 시점** | 초기(통합 문제를 1번 이슈에서 만남) | 말기(레이어 결합 시점에야 드러남) |
| **병렬 작업 용이성** | 수직 의존성 있어 한 명이 풀스택으로 가기 유리 | 레이어별 팀/역할 분리에 유리 |
| **이슈 크기 안정성** | 한 이슈에 여러 레이어가 섞여 커지기 쉬움 | 각 이슈가 레이어 단위라 크기 고르게 맞추기 쉬움 |
| **데모 가능성** | 매 이슈마다 데모 가능 | 레이어 이슈들이 묶여야 데모 가능 |
| **적합한 상황** | 신규 MVP, 프로토타입, 고객 피드백 루프가 중요한 제품 | 레거시 리팩터링, 인프라 재구축, 백/프론트 팀이 분리된 조직 |
| **주요 리스크** | 중복 구현, 설계 일관성 부족 | 통합 지연, "거의 다 됐는데 아직 아무것도 안 돌아감" 상태 |

## 같은 기능을 각 방식으로 쪼개면

**예시 기능**: "사용자가 할 일(Todo)을 추가하고 목록을 볼 수 있다."

### 수직 슬라이스로 분할하면

- `[Skeleton]` Hello World 페이지 배포 + Health API + DB 연결 ← **Walking Skeleton**
- `[Slice]` Todo 1건 추가 (Input UI + POST API + INSERT + 테스트)
- `[Slice]` Todo 목록 조회 (List UI + GET API + SELECT + 테스트) ← **이 시점부터 MVP**
- `[Slice]` 삭제 기능
- `[Ops]` 에러 모니터링 연동

> 1번 이슈가 끝나면 배포 URL이 존재하고, 3번 이슈만 완료돼도 "Todo 추가·조회"가 실제로 동작한다.
> "스케이트보드 → 킥보드 → 자전거"처럼 **매 단계가 이미 유용**하다.

### 아키텍처 계층별로 분할하면

- `[Setup]` 저장소·CI 구성
- `[DB]` Todo 테이블 마이그레이션
- `[Backend]` Todo CRUD API 전체
- `[Frontend]` Todo 입력·목록·삭제 컴포넌트
- `[Test]` 통합 테스트
- `[Docs]` API 레퍼런스

> 모든 이슈가 끝나야 배포 가능하다. "바퀴 → 차축 → 차체 → 자동차" 순서라 마지막 주까지 아무 것도 굴러가지 않는다.
> 대신 각 이슈의 담당자와 범위가 명확해서 팀 분업엔 편하다.

## 선택의 기준

### 수직 슬라이스를 선택하는 게 좋은 때

1. 배포 파이프라인조차 없는 신규 프로젝트
2. MVP를 빠르게 사용자에게 보여주고 피드백을 돌려야 하는 상황
3. 한 명~소수 풀스택 개발자가 주도하는 경우
4. 통합 리스크(CORS, 인증 토큰, 배포 구성 등)를 조기에 걸러내고 싶은 경우

> 요지: "매 이슈가 끝날 때마다 데모할 수 있어야 한다"는 원칙을 지키고 싶을 때.

### 아키텍처 계층별 분할이 낫는 때

1. 데이터 모델이 이미 확정된 레거시 시스템의 리팩터링
2. 백엔드·프론트엔드 팀이 조직적으로 분리되어 병렬 작업이 핵심인 경우
3. 대규모 인프라 재구축처럼 유저 스토리 단위로 쪼개는 것이 부자연스러운 작업
4. 각 레이어가 완성도 높은 스펙을 가지고 있어 계약만 맞추면 되는 경우

## 절충안 — 하이브리드

실무에서는 초반 1~2주는 수직 슬라이스로 Walking Skeleton과 핵심 MVP를 띄우고, 이후 Phase 3 운영화 단계 또는 특정 대규모 리팩터링 작업에 한해 계층별 분할로 넘어가는 하이브리드가 가장 흔하다.

두 스킬(`generate-issues-vertical`, `generate-issues-layered`)을 같은 저장소에서 섞어 쓰더라도 `strategy:vertical-slice` / `strategy:layered` 공통 레이블로 구분되므로 혼선 없이 병행 운영이 가능하다.

## 요약

| 우선시하는 가치 | 선택할 전략 |
|----------------|-------------|
| "사용자에게 가치가 언제 전달되는가" | **수직 슬라이스** |
| "레이어의 내부 완성도와 분업의 명확성" | **계층별 분할** |

그리고 수직 슬라이스 전략을 선택했다면 순서는 항상:

> **Walking Skeleton (Phase 0, 일단 뭐든 배포됨)** → **MVP (Phase 1~2, 사용자에게 가치 전달)** → **운영화 (Phase 3, 품질·관측성 강화)**
